架构方案对比

Three.js 与 React 的三种协作模式

方案 A：React 管控 Three.js

React owns state → Three.js is a dumb renderer
Zustand Store (所有状态)
    │
    ├── ▶ React App
    │       ├── ▶ <ThreeCanvas> (props: nodes, edges)
    │       │       └── 每帧从 store 读取渲染
    │       ├── ▶ <HexagramRadar> (props: stats)
    │       └── ▶ <SocialFeed> (props: posts)
    │
    └── SSE Event → dispatch → store 更新 → re-render

优势

  • React 开发者友好，调试方便
  • 状态流清晰，单向数据流

问题

  • Three.js 跟着 React re-render 走，粒子动画帧率可能不稳
  • 60fps 动画循环与 React 18 并发模式可能冲突

方案 B：Three.js 自成引擎，React 只做浮层

Three.js owns 3D world → React is overlay UI only
Three.js Engine (独立动画循环，独立场景状态)
    │
    ├── requestAnimationFrame 循环
    │   └── 控制所有 3D 渲染 + 粒子系统
    │
EventBus (双向桥)
    │
    ├── Three → React: "节点被点击" / "动画完成"
    └── React → Three: "生成分支" / "切换选项"

React Layer (HTML overlay, z-index 在 Canvas 上方)
    ├── ▶ <ChoiceWheel>
    ├── ▶ <HexagramRadar>
    └── ▶ <SocialFeed>

优势

  • Three.js 完全控制渲染，帧率保证
  • 粒子系统性能最优

问题

  • 两套状态各自维护，同步容易出 bug
  • React 组件不能直接读取 Three 内部状态

方案 C：混合模式 推荐

Zustand 单一真相源 + Three.js 本地动画状态
Zustand Store (唯一数据源)
    │  nodes[] / edges[] / rpgStats / socialPosts / uiState
    │
    ├── ▶ useGraphStore() (订阅)
    │       └── <ThreeCanvas>
    │             ├── useEffect: store 变化 → 更新 Three.js scene
    │             ├── requestAnimationFrame: 粒子 / 动画 / 力导向
    │             └── 事件回调 → store.dispatch() (例如: 点击节点)
    │
    ├── ▶ useRPGStore() → <HexagramRadar> (纯 React SVG)
    └── ▶ useSocialStore() → <SocialFeed> (纯 React DOM)

优势

  • Zustand 单一真相源，Three.js 和 React 都从同一个 store 读取
  • Three.js 用 useEffect 响应数据变化，用 rAF 独立跑动画
  • 粒子 / 六芒星 / 舆论三套 UI 互不干扰

代价

  • 需要精细控制 useEffect deps 避免不必要的 Three.js 重建
  • rAF 循环中不能直接读 React state（用 ref 桥接）
