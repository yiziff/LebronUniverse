## 1. 核心创意评估

### 1.1 灵魂所在

> "允许用户点击'取消这笔交易'，看着整条时间线坍塌、重组，生成一个全新的平行宇宙。"

这个交互模型比"静态可视化"高了一个维度。它把一个**展示型应用**变成了一个**沙盒引擎**。用户从旁观者变成了上帝。

### 1.2 创意质量判定

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 独特性 | ⭐⭐⭐⭐⭐ | 目前市面上不存在同类产品。Basketball Reference 有交易记录，TradeNBA 有交易模拟器，但**没有人做"删除历史"** |
| 传播性 | ⭐⭐⭐⭐⭐ | "如果快船没有换 PG"这类话题在篮球社区有极高的讨论度。demo 视频天然适合 Twitter/Reddit/虎扑 |
| 技术可行性 | ⭐⭐⭐⭐☆ | 算法核心是 DAG 遍历 + LLM API 调用，不涉及 ML 训练、分布式系统或高并发 |
| 扩展性 | ⭐⭐⭐⭐⭐ | 可以从 5 笔交易开始，逐步覆盖 NBA 全历史。每个新交易都是一条新内容管线 |
| 商业化潜力 | ⭐⭐⭐☆☆ | 更多是 portfolio piece / 技术博客素材，直接变现路径有限但间接价值高 |

**这是整个项目的第一大瓶颈**。NBA 交易的复杂度远超表面：

- **选秀权保护条件**：前 3 保护、乐透保护、逐年递减保护、转换为次轮、转换为未来首轮……每个保护条件都是一个状态机。

> **实战策略：渐进式覆盖**

> - MVP 阶段：只录入 **5-8 笔**关键交易（PG-SGA、Harden 去篮网、Gobert 去森林狼、AD 去湖人、Mitchell 去骑士），手工确保数据 100% 准确。

#### 核心逻辑

```
用户点击"取消 PG-SGA 交易"
           │
           ▼
┌──────────────────────┐
│ 标记该交易为已取消     │
│ 其流出资产全部"回滚"   │
│ PG → 回雷霆           │
│ SGA → 回快船          │
│ 选秀权 → 回快船        │
└──────────────────────┘
           │
           ▼
┌──────────────────────────────────────┐
│ BFS 遍历下游交易图                     │
│                                      │
│ 对于每一个使用了"已被剪枝资产"的交易：   │
│   → 如果该交易的所有前置资产都已失效：   │
│       标记该交易为无效                 │
│       递归剪枝其下游                    │
│   → 如果仅部分前置资产失效：            │
│       [设计决策] 标记为"部分有效"        │
│                  或"整体无效"          │
└──────────────────────────────────────┘
           │
           ▼
┌──────────────────────┐
│ 示例：                │
│ 剪枝 PG-SGA →        │
│   ├─ 雷霆用快船2022    │
│   │  首轮选中 J-Dub    │
│   │  标记为"从未发生"   │
│   │                   │
│   ├─ 雷霆用快船2024    │
│   │  首轮选中 Topic    │
│   │  标记为"从未发生"   │
│   │                   │
│   └─ 雷霆后续用这些     │
│      筹码做的所有交易   │
│      全部坍塌           │
└──────────────────────┘
```

#### 伪代码实现

```python
def cascade_prune(graph: TradeGraph, cancelled_trade_id: str) -> PruneResult:
    """
    级联剪枝：切断一条交易边后，递归剪除所有依赖该交易的下游节点。
    
    时间复杂度：O(V + E)，即图的大小。
    因为每个节点和边最多被访问一次。
    """
    invalid_trades: set[str] = {cancelled_trade_id}
    invalid_assets: set[str] = set()
    queue: deque[str] = deque([cancelled_trade_id])
    
    # 第一步：收集被取消交易的流出资产
    cancelled = graph.get_trade(cancelled_trade_id)
    invalid_assets.update(cancelled.assets_out)
    
    # 第二步：BFS 遍历下游
    while queue:
        trade_id = queue.popleft()
        
        for downstream_id in graph.get_downstream_trades(trade_id):
            if downstream_id in invalid_trades:
                continue  # 已处理过
                
            downstream = graph.get_trade(downstream_id)
            
            # 关键判断：这笔下游交易的所有输入资产是否都来自已失效的交易？
            all_ancestors_invalid = all(
                asset.source_trade_id in invalid_trades
                for asset in downstream.assets_in
            )
            
            if all_ancestors_invalid:
                invalid_trades.add(downstream_id)
                invalid_assets.update(downstream.assets_out)
                queue.append(downstream_id)
    
    return PruneResult(
        invalid_trades=invalid_trades,
        invalid_assets=invalid_assets,
        affected_teams=derive_affected_teams(invalid_trades)
    )
```

#### 需要处理的关键边界情况

| 场景 | 处理策略 | 难度 |
|------|---------|:----:|
| **部分资产失效**：一笔交易涉及 5 个资产，只有 1 个来自被剪枝交易 | MVP 阶段统一标记为"整体无效"，因为现实中交易也不可能"部分发生" | 低 |
| **选秀权→球员转化**：快船 2022 首轮被送到了雷霆，雷霆选中 J-Dub | 需要在图中建模"选秀大会"节点作为中间态 | 中 |
| **循环依赖**：A 队把球员送到 B 队，B 队后来又把同一个球员送回 A 队 | 打上 `source_trade_id` 就能追踪；循环不影响剪枝逻辑，每个节点只看祖先 | 低 |
| **独立分支**：一笔交易与取消的交易完全无关 | 不受影响，继续显示为金色"真实历史"线 | 无 |
| **选秀权互换权**：快船和雷霆互换 2025 首轮 | 互换权本身是一个"选择权"节点，剪枝时需特殊处理 | 中 |

#### 3.3.1 Prompt 工程设计

LLM 需要三样东西才能生成可信的平行宇宙：

1. **当前世界状态**（剪枝后各队的资产清单）
2. **历史锚点**（真实历史中那些资产后来怎么样了）
3. **约束条件**（薪资帽规则、当时自由市场名单、球队管理层交易风格）

```
┌─────────────────────────────────────────────────┐
│              SYSTEM PROMPT                       │
│                                                  │
│  你是 NBA 平行宇宙推演引擎。                       │
│  你的每一次输出都会产生一个全新的时间线分支。       │
│                                                  │
│  【推演规则】                                     │
│  1. 交易必须符合当时的薪资帽规则（硬工资帽逻辑）     │
│  2. 选秀权交易必须遵守 Stepien Rule               │
│  3. 自由球员签约需考虑球队薪资空间                  │
│  4. 交易价值评估需参考球员当时的年龄、合同、表现     │
│  5. 考虑球队总经理的交易风格                       │
│     （Presti 爱攒选秀权，Ainge 精于算计，等等）     │
│  6. 输出必须是严格的 JSON Schema                   │
│                                                  │
│  【当前世界状态】                                  │
│  {world_state_json}                              │
│                                                  │
│  【真实历史对照（这些资产后来发生了什么）】          │
│  {real_history_context}                          │
│                                                  │
│  【当时自由市场/交易市场上的主要球员】               │
│  {free_agent_context}                            │
│                                                  │
│  【球队状态摘要】                                  │
│  {team_contexts}                                 │
└─────────────────────────────────────────────────┘
```

#### 3.3.3 质量保证机制

| 机制 | 说明 | 实现方式 |
|------|------|---------|
| **Schema 校验** | 确保 LLM 返回的 JSON 符合预期结构 | Pydantic / Zod validation |
| **实体校验** | 确保 LLM 没有"发明"球员 | 球员名单白名单比对 |
| **薪资校验** | 确保交易薪资匹配 | 硬工资帽规则引擎（简单版） |
| **重试机制** | LLM 输出不合格时重新生成 | 最多 3 次，每次用不同种子 |
| **用户裁判** | 最终由用户决定"这个平行宇宙是否合理" | 前端提供"合理/不合理"投票 |

```
真实历史线：  ════════ ●════════ ●════════ ●═══════
（金色/暖色）         （交易A）  （交易B）  （交易C）

平行宇宙分支：           ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ →
```

#### 3.4.2 推荐技术栈

| 库 | 适用场景 | 优势 |
|---|---------|------|
| **D3.js** | 自定义力导向图、时间线 | 最强的灵活性，但学习曲线陡 |
| **vis.js / vis-network** | 开箱即用的网络图 | 上手快，API 友好 |

| **GSAP** | 动画 | 节点坍塌/重连动画的终极方案 |

#### 3.4.3 关键交互

1. **悬停节点** → 显示交易详情弹出框
2. **点击边上的 × 按钮** → "取消此交易"确认对话框
3. **确认后动画** → 下游节点逐级破碎（粒子效果），时间线收缩
4. **Loading** → "正在推演平行宇宙……"（LLM API 可能需要 3-10 秒）
5. **新分支出现** → 霓虹绿色线条从缺口处生长出来
6. **嵌套取消** → 用户可以在 AI 生成的交易上再次点击 ×，产生二级平行宇宙

## 4. 最小可行产品定义

### 4.1 MVP 范围

**一句话**：5 笔种子交易 + 1 个蝴蝶效应入口 + AI 生成 1 层平行宇宙。

| 包含 | 不包含 |
|------|--------|
| PG-SGA 互换（2019） | 全 NBA 历史覆盖 |
| Harden 去篮网（2021） | 三方/四方交易 |
| Gobert 去森林狼（2022） | 选秀权保护条件的完整覆盖 |
| AD 去湖人（2019） | 薪资帽规则引擎 |
| Mitchell 去骑士（2022） | 用户账户 / 持久化 |
| 单击"取消" → 剪枝 → AI 生成 1 条新分支 | 多级嵌套取消（可以后续加） |
| 两种颜色的力导向图 | 移动端适配 |

### 4.2 MVP 验收标准

- [ ] 用户可以看到 5 笔交易的金色时间线图
- [ ] 点击任意交易的 × 按钮，下游节点破碎动画正常播放
- [ ] 剪枝后的空白被一条霓虹绿新分支填充（AI 生成）
- [ ] 新分支的 JSON 数据 schema 合法
- [ ] 刷新页面后数据不丢失

## 5. 技术栈建议

### 5.1 后端：Python（推荐）

| 组件 | 选择 | 理由 |
|------|------|------|
| Web 框架 | **FastAPI** | 异步支持好、自动生成 OpenAPI 文档、Pydantic 天然适配数据校验 |
| 图数据结构 | **NetworkX** | Python 最成熟的图算法库，内置 BFS/DFS/拓扑排序 |
| LLM SDK | **Anthropic SDK** 或 **OpenAI SDK** | 取决于你用什么模型。Claude 在结构化 JSON 输出上更稳定 |
| 数据存储 | **SQLite + JSON files** | MVP 阶段不需要 PostgreSQL。种子数据直接存 JSON，生成的分支存 SQLite |
| Schema 校验 | **Pydantic v2** | 和 FastAPI 无缝集成，性能好 |

### 5.2 后端：Node.js / TypeScript（备选）

| 组件 | 选择 | 理由 |
|------|------|------|
| Web 框架 | **Hono** | 轻量、快、TypeScript 原生 |
| 图数据结构 | **graphlib** 或手写 | JS 生态没有 NetworkX 级别的图库 |
| Schema 校验 | **Zod** | TypeScript 第一公民 |

### 5.3 前端

| 组件 | 选择 | 理由 |
|------|------|------|
| 框架 | **React + TypeScript** | 生态最丰富 |
| 图可视化 | **React Flow** | 专为 React 设计的节点图库，交互开发体验极好 |
| 动画 | **Framer Motion** | 声明式动画，适合节点入场/退场 |
| UI 组件 | **shadcn/ui** | 现代化、可定制 |
| 构建工具 | **Vite** | 快 |

## 6. 时间与人力估算

> 单人开发，每周投入约 20-25 小时（业余时间）。

| 阶段 | 时间 | 关键产出 | 风险 |
|------|:--:|------|:--:|
| **Phase 1: 数据建模** | 3-5 天 | Player / DraftPick / Trade 模型 + 5 笔种子数据 JSON | 数据不准确导致后续一切无效 |
| **Phase 2: 级联剪枝引擎** | 2-3 天 | `cascade_prune()` + 单元测试 + 可视化调试工具 | 边界情况遗漏 |
| **Phase 3: AI 推演引擎** | 3-5 天 | Prompt 模板 + Schema 校验 + 异步任务队列 | LLM 输出质量不稳定 |
| **Phase 4: 前端可视化** | 3-5 天 | 力导向图 + 交易面板 + 蝴蝶效应按钮 + 动画 | 性能（节点太多时） |
| **Phase 5: 联调与打磨** | 2-3 天 | 端到端流程跑通 + demo 视频录制 | 集成问题 |
| **MVP 总计** | **13-21 天** | 一个会让你想炫耀的项目 | — |
| **Phase 6: 扩展到全量数据** | 持续 | 每增加一笔交易 = 30-60 分钟数据录入 + 测试 | 苦力活 |

## 7. 风险与陷阱

### 🔴 高风险

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| **NBA 交易数据获取困难** | 数据层瘫痪 | MVP 只做 5 笔，手工录入。后续考虑爬取 Basketball Reference 或用 LLM 辅助提取 |
| **选秀权保护条件极端复杂** | 无法精确建模 | MVP 阶段忽略保护条件，假设所有选秀权都"传达"了。后续逐步添加 |

### 🟡 中风险

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| **LLM 生成结果不稳定** | 平行宇宙质量参差不齐 | 多次生成 + Schema 校验 + 实体白名单 + 用户反馈机制 |
| **图可视化性能** | 节点超过 100 个时可能卡顿 | MVP 节点数控制在 50 以内。后续可以用虚拟化 + WebGL |
| **LLM 幻觉（发明不存在的人）** | 破坏可信度 | 球员/球队白名单硬约束 + 输出后自动过滤 |

### 🟢 低风险

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| **前端动画复杂** | UI 不够流畅 | 降低动画复杂度，优先保证功能完整 |
| **LLM API 成本** | 每次推演约 $0.01-0.10 | 可忽略不计。Claude Haiku 极便宜且推理能力足够 |

## 8. 为什么这个项目值得做

### 8.1 作为 Portfolio Piece

这个项目在一个 demo 视频里展示了：
