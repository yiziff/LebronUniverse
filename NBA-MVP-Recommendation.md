# 🏀 先做哪个 MVP？— 推荐分析

> **核心问题**：交易蝴蝶效应（PG-SGA） vs 詹姆斯 2010 决定，从哪个起手？
>
> **结论前置**：**从交易蝴蝶效应起手。但两个项目其实是同一个项目。**

---

## 1. 两项目对比矩阵

| 维度 | PG-SGA 交易蝴蝶效应 | 詹姆斯 2010 决定 |
|------|:--|:--|
| **核心交互** | 点击 × 取消一笔交易 | 在轮盘中选择一个去向 |
| **核心技术** | DAG 级联剪枝算法（BFS） | Prompt 工程 + 流式生成 |
| **AI 角色** | 辅助（填补剪枝后的空白） | 主导（构建整条世界线） |
| **代码复杂度** | ★★★★★☆ (算法重) | ★★★☆☆☆ (集成重) |
| **数据录入量** | 1 笔交易 + 下游链条 ≈ 3h | 5 支球队状态 ≈ 4h |
| **MVP 周期** | 7-10 天 | 7-10 天 |
| **demo 视觉冲击** | 坍塌动画（极强） | 分支生长（强） |
| **面试算法展示** | DAG / BFS / 图遍历 | 较弱 |
| **面试工程展示** | 数据建模 + 状态管理 | Prompt 工程 + 流式架构 |
| **篮球迷传播力** | "如果快船没换 PG 会怎样" | "如果詹姆斯去了尼克斯" |

## 2. 为什么推荐从 PG-SGA 交易起手

### 2.1 原因一：算法能力是面试硬通货

```
PG-SGA 项目在面试中的对话：
面试官："你做的最有技术深度的项目是什么？"
你："我做了一个 NBA 历史交易的有向无环图。
     用户可以取消任意一笔交易，我写了一个级联剪枝算法，
     基于 BFS 递归剪除所有依赖该交易的后续节点。
     比如取消 PG-SGA 交易，雷霆用快船首轮选中的
     J-Dub 节点会自动失效。"
面试官内心："这人真的写过一个非平凡的图算法。不是调包侠。"

────────────────────────────────────────────────

詹姆斯 2010 项目在面试中的对话：
面试官："你做的最有技术深度的项目是什么？"
你："我做了一个 AI 驱动的 NBA 多重宇宙推演引擎，
     用户可以替詹姆斯选择不同球队，AI 会生成平行世界线。"
面试官内心："Prompt 工程 + API 调用。有创意但技术深度一般。"
```

**对于"找实习、过简历关"这个真实目标，PG-SGA 的算法展示价值更高。**

### 2.2 原因二：PG-SGA 的 demo 视觉更有破坏力

```
PG-SGA demo 的 90 秒：

  0-10s   金色时间线，5 个交易节点，安静展示
  10-15s  用户 hover 到 PG-SGA 节点，弹出 × 按钮
  15-20s  用户点击 ×
  20-25s  确认对话框："这将导致 8 个下游节点坍塌"
  25-35s  坍塌动画开始 —— 节点逐个炸裂，粒子向屏幕外飞散
          J-Dub → 碎掉
          Topic → 碎掉  
          Caruso交易 → 碎掉
          时间线在 2019 年处断开，留下一个脉动的裂口
  35-45s  "正在推演平行宇宙……" 裂口中霓虹绿光脉动
  45-80s  新节点从裂口中逐个生长出来
          绿色新世界线展开
  80-90s  双线同屏：金色残骸 vs 霓虹新世界
  
  这一套动画没有任何篮球知识也能看懂。
  面试官不需要认识 SGA。
  他只需要看到"一条链子像多米诺骨牌一样坍塌了"。

────────────────────────────────────────────────

詹姆斯 2010 demo 的 90 秒：

  0-10s   金色时间线，2010 分叉点发光
  10-20s  点击 → 轮盘弹出，4 个选项
  20-30s  选择尼克斯 → 金色线裂开
  30-80s  AI 生成中，节点逐个出现（等待感较强）
  80-90s  两条线并排显示
  
  冲击力主要来自"AI 生成了什么内容"，而非"画面发生了什么"。
```

**PG-SGA 的视觉冲击力来自于破坏和重建的动画过程。詹姆斯项目的冲击力依赖于用户阅读 AI 生成的内容。前者是电影，后者是小说。**

### 2.3 原因三：PG-SGA 的图数据结构是詹姆斯项目的底层基础

这是最关键的技术原因：

```
PG-SGA 项目建立的数据结构：

  Trade {
    id, timestamp, assets_in, assets_out,
    source_trade_ids,    ← 资产溯源
    downstream_trade_ids  ← 下游依赖
  }

  这个结构 + 级联剪枝算法 = 一个可用的交易图谱。


詹姆斯项目需要的底层：

  分叉点 → 用户选择 → AI 生成新交易 → 新交易之间也有上下游关系
  → 用户在 AI 生成的交易上也可能想点 ×
  
  这不就是同一个数据结构吗！


结论：
  做完 PG-SGA = 你已经有了一套能跑的 DAG 引擎。
  往上加"分叉点"只是新增一个节点类型，不需要改任何底层逻辑。
  两个项目天然是同一个项目的两个 phase。
```

### Phase 1 和 Phase 2 共享的代码

```

  ✅ Trade 模型 (Pydantic)
  ✅ 球员 / 选秀权 / 球队模型
  ✅ DAG 图结构（NetworkX）
  ✅ 级联剪枝算法
  ✅ LLM 调用 + Schema 校验
  ✅ 流式返回（SSE）
  ✅ React Flow 力导向图
  ✅ 双色时间线渲染
  ✅ 节点悬停详情卡片
  ✅ localStorage 缓存

Phase 2 只需要新增：
  ➕ DecisionEvent 模型（继承自 Trade 的基础结构）
  ➕ 抉择轮盘 UI 组件
  ➕ 世界线生成 Prompt 模板
  ➕ 詹姆斯 2010 数据 JSON
  ➕ 分支切换逻辑
```

**Phase 1 的 70% 代码在 Phase 2 中直接复用。这不是两个项目，是一个项目的两次迭代。**

## 4. 为什么不做"纯詹姆斯项目"作为起点

如果从詹姆斯决定起手，你会面临一个尴尬的境地：

```
詹姆斯项目做完后：

  问：用户能在 AI 生成的交易上点 × 吗？
  答：不行，还没做剪枝逻辑。
  
  问：那用户怎么"撤销"AI 生成的某个事件？
  答：没办法，只能重新选。
  
  问：这不就是一个"选择 → 看 AI 输出"的一次性体验吗？
  答：……是。
```

詹姆斯项目如果没有剪枝能力，就像一个只能读不能写的 Git。用户选了一个选项，AI 生成一条线，结束。没有进一步的交互。而 PG-SGA 做完后，用户可以在**任何节点**上点 ×——包括 AI 生成的节点——这是一个无限可玩的沙盒。

## 附：如果你仍然想先做詹姆斯项目

如果你觉得詹姆斯项目更让你兴奋（这是合理的，兴奋感 > 技术合理性），那就做詹姆斯项目。但要做一个小改动：

```
在詹姆斯项目的 MVP 中，仍然先实现 Trade 数据模型。

因为 AI 生成的内容就是交易和事件，
这些交易和事件天然需要 Trade 模型来存储和渲染。
你不可避免要写这个模型。

所以即使从詹姆斯起手，
你也会在 Day 1-2 先写完 Trade + Player + DraftPick 模型。
这和 PG-SGA 项目的 Day 1-2 是一模一样的。

两条路在前两天是重合的。
第三天开始分叉。
```

---

> **一句话：从 PG-SGA 开始。做完后别停，顺手把詹姆斯加进去。两个 MVP 合计 3-4 周，做出一个在面试和 GitHub 上都能打的完整项目。**
