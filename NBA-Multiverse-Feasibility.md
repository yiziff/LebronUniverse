## 1. 产品格局：从"撤销键"到"多重宇宙"

### 1.1 两个版本的对比

| 维度 | v1.0 交易蝴蝶效应 | v2.0 多重宇宙漫游 |
|------|:--|:--|
| **核心交互** | 点击 × 取消一笔交易 | 在历史分叉点**主动选择**走向 |
| **用户角色** | 旁观者 → 破坏者 | 上帝 → 造物主 |
| **体验隐喻** | "删除历史" | "Choose Your Own Adventure" |
| **分支深度** | 一层（取消 → AI填充） | 无限嵌套（选择 → 分支 → 再选择 → 再分支） |
| **传播力** | "看这个NBA交易模拟器" | "如果詹姆斯去了尼克斯，NBA会怎样？" |
| **面试吸引力** | 算法 + 全栈 + 创意 | 图论 + 状态机 + LLM工程 + 产品设计 |

v2.0 不是 v1.0 的增量升级。**这是一个完全不同的产品。**

### 1.2 为什么 v2.0 值得做

v1.0 让面试官觉得"这人技术不错"。
v2.0 让面试官觉得"这人有产品格局"。

区别在于：v1.0 展示的是**执行能力**，v2.0 展示的是**从零构想的架构能力**。对于一个实习候选人的简历来说，后者是核弹级别的。

## 3. 技术架构：Git 分支模型

### 3.1 核心理念

真实历史是一条 `master` 分支。每一次用户的选择 = 一次 `git checkout -b <new-branch>`。

```
                    ○ 2011东决G7胜凯
                   ╱
    ○ 詹姆斯去热火  ──→  ○ 2012总冠军  ──→  ○ 2013总冠军  ──→  ○ 2014败马刺  ← master（真实历史）
   ╱
  ● 2010年夏天
   ╲
    ○ 詹姆斯去尼克斯  ──→  ○ 2011东决  ──→  ○ 2012败热火  ──→  ○ 2013交易得甜瓜
                   ╲
                    ○ 2014再败  ──→  ● 2014杜兰特自由球员（新分叉点）
                                      ├── ○ 杜兰特加盟尼克斯
                                      └── ○ 杜兰特留守雷霆
```

### 3.2 Git 语义映射

```
┌──────────────────┬──────────────────────────────────────┐
│ Git 概念          │ NBA 多重宇宙映射                      │
├──────────────────┼──────────────────────────────────────┤
│ commit           │ 一个事件（交易/签约/选秀/伤病/季后赛结果） │
│ branch           │ 一条世界线（从分叉点到最后一个事件）      │
│ HEAD             │ 用户当前正在观看的世界线                │
│ checkout         │ 切换到另一条世界线                      │
│ merge            │ （暂不支持，事件不可合并）               │
│ rebase           │ （暂不支持）                           │
│ commit hash      │ event_id: SHA256(parent_id + choice)  │
│ DAG              │ 所有世界线构成的有向无环图               │
└──────────────────┴──────────────────────────────────────┘
```

### 3.3 后端存储结构

```
data/
├── seeds/
│   ├── events/
│   │   ├── event_001_lebron_2010_free_agency.json    # 分叉点定义
│   │   ├── event_002_pg_sga_trade_2019.json           # 分叉点定义
│   │   └── ...
│   ├── players.json                                    # 球员字典
│   └── teams.json                                      # 球队字典
│
└── runtime/
    └── multiverse.db   # SQLite
        ├── branches (branch_id, parent_event_id, choice, created_at)
        ├── events (event_id, branch_id, timestamp, type, description, state_diff)
        └── league_states (branch_id, event_id, team_id, roster_json, cap_space, picks_json)
```

## 5. 状态管理与上下文切换

### 5.1 核心挑战

```
用户在 2010 年选择 "詹姆斯去尼克斯"，观看 AI 推演出的 5 年新历史。
然后用户点击"回到分叉点"，选择 "詹姆斯去公牛"。
此时系统需要：
  1. 把纽约宇宙的全部推演节点从渲染层卸载
  2. 加载公牛宇宙（如果已生成）或触发新的 AI 推演
  3. 联盟状态（薪资、阵容、选秀权）瞬间恢复到 2010 年夏天
```

### 5.2 解决方案：状态快照链

```
                    snapshot_2010_summer_clean
                    (决策前的纯净状态)
                            │
          ┌─────────────────┼─────────────────┐
          │                 │                 │
    snapshot_miami      snapshot_nyk      snapshot_chi
    (热火三巨头)         (詹+小斯)          (詹+罗斯)
          │                 │                 │
          ├── 2011 决赛     ├── 2011 东决      ├── 2011 东决击败热火
          ├── 2012 冠军     ├── 2012 败         ├── 2012 冠军
          └── ...           └── ...            └── ...
```

每个 `snapshot` 是一个完整且不可变的球队状态快照。切换分支 = 加载对应的快照 + 应用后续事件。

### 5.3 实现思路

```python
class MultiverseEngine:
    """
    多重宇宙状态管理器。
    每个分支维护一条事件链，状态快照在分叉点处不可变。
    """
    
    def switch_branch(self, branch_id: str) -> LeagueState:
        """
        上下文切换的核心方法。
        1. 找到该分支的根快照
        2. 以快照为起点，执行事件链上的每个事件
        3. 返回当前联盟状态
        """
        branch = self.store.get_branch(branch_id)
        root_snapshot = self.store.get_snapshot(branch.parent_event_id)
        state = LeagueState.from_snapshot(root_snapshot)
        
        for event in branch.events:
            state = self.apply_event(state, event)
        
        return state
    
    def apply_event(self, state: LeagueState, event: Event) -> LeagueState:
        """
        将一个事件应用到当前联盟状态上。
        事件可能是人员变动、选秀、薪资调整等。
        """
        new_state = state.clone()
        for team_id, changes in event.state_changes.items():
            new_state.teams[team_id].apply(changes)
        return new_state
    
    def fork(self, parent_event_id: str, choice_id: str) -> Branch:
        """
        在某个分叉点创建新分支。
        如果该分支已存在（用户之前选过），直接返回缓存。
        否则触发 AI 推演。
        """
        existing = self.store.find_branch(parent_event_id, choice_id)
        if existing:
            return existing
        return self.generate_branch(parent_event_id, choice_id)
```

## 6. LLM 角色升级：从填补者到世界构建者

### 6.1 角色变化

| 维度 | v1.0 | v2.0 |
|------|------|------|
| LLM 工作 | 填补一条被剪断的时间线 | 从零构建一条完整的世界线 |
| 输出量 | 1-3 笔交易 | 5-10 个事件，覆盖 3-7 年 |
| 上下文 | 两支球队的状态 | 全联盟 30 支球队的连锁反应 |
| 一致性要求 | 局部的交易合理性 | 跨年份的叙事一致性 |

### 6.2 两阶段生成策略

```
Phase 1: 宏观叙事生成 (Macro Narrative)

  Prompt: "詹姆斯去尼克斯之后，接下来 5 年联盟格局如何变化？"
  
  LLM 输出：
  {
    "narrative_arc": "尼克斯在詹姆斯的带领下迅速成为东部豪强……",
    "major_plot_points": [
      { "year": 2011, "event": "尼克斯东部半决赛败给凯尔特人" },
      { "year": 2012, "event": "尼克斯交易得到卡梅隆·安东尼" },
      { "year": 2013, "event": "尼克斯赢得东部冠军，总决赛败给雷霆" },
      { "year": 2014, "event": "杜兰特决定加盟尼克斯" }
    ],
    "teams_significantly_affected": ["NYK", "MIA", "CLE", "CHI", "OKC"]
  }

─────────────────────────────────────────────────────

Phase 2: 逐事件精细化生成 (Per-Event Generation)

  以 Phase 1 的宏观叙事为骨架，逐事件让 LLM 生成：
  - 具体的交易/签约细节
  - 涉及的资产流动
  - 对全联盟状态的影响
  
  每个事件独立生成，但共享 Phase 1 的宏观叙事作为上下文约束。
```

### 6.3 Prompt 工程

```
SYSTEM PROMPT（约 3000 tokens）

你是 NBA 多重宇宙推演引擎。你的每一次输出都会生成一个完整的平行世界。

【硬约束】
1. 遵守 NBA 劳资协议（CBA），交易薪资必须匹配（±125%）

3. 自由球员签约必须有足够的薪资空间
4. 球员交易价值参考其当时年龄、合同、表现、伤病史
5. 球队行为需符合总经理历史风格：
```

## 9. 难度矩阵

| 模块 | 技术难度 | 工作量 | 风险 | 备注 |
|------|:---:|:---:|:---:|------|
| **分叉节点数据建模** | 6/10 | 3/10 | 🟡 | 需要想清楚 schema，一旦定型后面都基于此 |
| **状态快照系统** | 7/10 | 5/10 | 🟡 | 核心挑战：不可变快照 + 事件应用 + 上下文切换 |
| **Git 分支模型存储** | 5/10 | 4/10 | 🟢 | SQLite 三张表，逻辑清晰 |
| **LLM 宏观叙事生成** | 7/10 | 5/10 | 🔴 | Prompt 工程量大，需要大量调试 |
| **LLM 逐事件生成** | 6/10 | 4/10 | 🟡 | 在宏观叙事约束下生成，一致性更高 |
| **LLM 输出校验** | 5/10 | 3/10 | 🟡 | Schema 校验 + 实体白名单 + 薪资合理性 |
| **抉择轮盘 UI** | 5/10 | 3/10 | 🟢 | 前端视觉效果，技术不复杂 |
| **力导向图可视化** | 6/10 | 4/10 | 🟡 | D3.js / React Flow，节点多了需要优化 |
| **分支切换动画** | 4/10 | 3/10 | 🟢 | Framer Motion / GSAP |
| **全联盟状态模拟** | 8/10 | 6/10 | 🔴 | 30 队的精确状态模拟是地狱级体力活 |

### 最难的三个问题

**1. LLM 世界线内部自洽性（难度 8/10）**

```
问题：AI 生成了"2011年尼克斯进东决"。
      后续事件中 AI 又说"2011年尼克斯乐透签选中某某"。
      这是自相矛盾的——进了东决的球队不可能有乐透签。
      
解决：两阶段生成 + 交叉验证。
      Phase 1 生成宏观叙事作为"不可动摇的骨架"。
      Phase 2 在骨架上填肉，每次填肉前把骨架作为约束传给 LLM。
      生成完成后，用一个轻量级校验 LLM 调用做一致性检查。
```

**2. 状态快照的精确性（难度 7/10）**

```
问题：2010年夏天，联盟中每支球队的具体阵容、合同、选秀权情况。
      这个数据量巨大，且需要精确。
      
解决：MVP 阶段只精确建模"受影响的球队"。
      比如詹姆斯决定只精确建模 NYK/MIA/CLE/CHI。
      其他 26 队用"基线状态"（真实历史中的状态）。
      后续逐步扩展覆盖。
```

**3. 链式 LLM 调用的延迟（难度 5/10）**

```
问题：生成一条完整世界线需要 5-10 次 LLM 调用。
      每次 2-5 秒，总延迟 30-60 秒。

解决：
  - 流式返回：每生成一个事件就立即渲染，不等全部完成
  - 并行生成：独立球队的事件可以并行调用 LLM
  - 预生成：热门选项（詹姆斯去尼克斯、杜兰特去凯尔特人等）预生成并缓存
  - 进度条：用生动的文案替代枯燥的百分比
    "正在推演2012年东部格局……"
    "正在计算韦德的下一站……"
    "正在模拟2013年选秀大会……"
```
