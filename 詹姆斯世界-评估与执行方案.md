## 2. 核心架构升级评估

### 2.1 从 2D → 3D：这是对的决定吗？

```
【当前 2D React Flow 方案】
  优点：
  - 开发速度快（React 原生生态）
  - 节点交互（hover/click/drag）开箱即用
  - 面试展示"工程能力"足够
  
  缺点：
  - 视觉冲击力天花板低
  - 无法实现粒子消散/聚合效果
  - "看起来像是公司内部工具"

─────────────────────────────────────

【新构想 3D Three.js 方案】
  优点：
  - 视觉冲击力拉满（面试官打开页面的第一秒就被震住）
  - 粒子消散/聚合 = 核心"名场面"
  - 星团引力连线 = 天然符合"多重宇宙"概念
  - 市面上没有任何人做过

  缺点：
  - 开发时间是 2D 的 2-3 倍
  - 节点交互需要手写（没有 React Flow 的便利）
  - 性能调优更复杂
```

**判定：3D 方案做。** 理由很简单——2D 方案能让你过简历关，3D 方案能让面试官主动找你聊天。这是一个投资回报率极高的决策。

### 2.2 推荐 3D 技术栈

| 层级 | 选择 | 理由 |
|------|------|------|
| 3D 引擎 | **Three.js** (vanilla) | 灵活度最高，生态最成熟 |
| 3D 图布局 | **3d-force-graph** | 基于 Three.js 的力导向图封装，直接支持节点/连线 3D 渲染 |
| 粒子系统 | **Three.js Points + ShaderMaterial** | 自写着色器实现粒子消散/聚合 |
| 3D UI 叠加 | **Html 组件覆盖在 Canvas 上** | React 组件层叠在 Three.js canvas 上方 |
| 动画编排 | **GSAP** | 粒子动画的 timeline 控制比 Framer Motion 更适合 3D 场景 |

## 4. 技术选型重新论证

### 4.1 为什么继续用 DeepSeek 而不是 Claude？

当前代码已在用 DeepSeek API（`deepseek-chat`）。对于这个项目，DeepSeek 有两个关键优势：

```
1. 成本：DeepSeek 的 API 价格约为 Claude 的 1/10。
   这个项目每次生成要 4096 tokens，用户会反复玩。
   用 Claude = 每次几毛钱，用 DeepSeek = 每次几分钱。

2. 中文 NBA 知识：DeepSeek 对中文 NBA 内容的训练覆盖很好。
   生成的中文篮球叙事质量不低于 Claude。

但有一个风险：JSON 格式遵循度。
DeepSeek 在严格 JSON Schema 输出上不如 Claude 稳定，
但你的 llm_engine.py 已经实现了流式 JSON 提取 + fallback 解析，
这个工程化缓冲层可以弥补。
```

**建议：继续用 DeepSeek，但保留切换到 Claude 的能力（接口已抽象）。**

### 4.2 为什么用 3d-force-graph 而不是手写 Three.js？

```
3d-force-graph 提供了：
  - 开箱即用的 3D 力导向布局
  - 节点/连线的事件系统 (onNodeClick, onNodeHover)
  - 节点颜色/大小/标签自定义
  - 与 Three.js 场景的互操作能力

你可以在它的基础上：
  - 注入自定义 ShaderMaterial 实现粒子效果
  - 操作底层 Three.js 场景添加星云背景
  - 在力导向引擎以外添加手动定位的节点
  
这比手写节省至少 2 周。
```

## 附 B: 关于"玩家同时操控两人 vs 扮演一人"的设计建议

你在文档末尾提出了一个非常好的设计问题。以下是分析：

```
【方案 A：同时操控两人】
  优点：交互深度最大，"上帝模式"体验
  缺点：复杂度翻倍，用户认知负担大，"我到底在玩谁？"
  适合：硬核 NBA 球迷

【方案 B：扮演一人，另一人由 AI 自动演化】
  优点：叙事聚焦清晰，用户代入感强
  缺点：另一人的故事线用户无法控制
  适合：大多数用户

【推荐方案：方案 B 为主 + "窥视"模式】
  默认：用户扮演 LeBron，KD 作为不可控的 NPC 世界变量
  但：用户在任何时候可以点击 KD 的星团
      → 弹出"窥视面板"
      → 展示 KD 在当前宇宙中的 AI 生成人生轨迹
      → 显示关键节点："在这个宇宙，KD 2016年选择了……"
      → 有一个小按钮：[🔀 以KD视角继续]
      → 点击后切换到扮演 KD

  这样：
  - 单一时刻用户只扮演一人（减少认知负担）
  - 但你保留了两条命运线的"纠缠感"
  - 用户可以在两条线之间切换
  - 每次切换就是一个全新的故事视角
```
